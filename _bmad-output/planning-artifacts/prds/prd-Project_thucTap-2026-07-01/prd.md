# PRD: CV Ingestion Service

## 0. Document Purpose

This Product Requirements Document (PRD) defines the functional and non-functional requirements for the **CV Ingestion Service**, a core microservice of the automated CV screening system. It is written for developers, architects, and QA engineers who will implement, test, and validate this service. 

This PRD establishes the semantic vocabulary (Glossary), core REST API contracts, RabbitMQ payload schemas, and system resilience mechanisms required to manage high-throughput file ingestion and downstream processing backpressure.

## 1. Vision

The CV Ingestion Service is the entry point of the screening platform. It aims to ingest resume files (PDF/DOCX) at scale, validate them, store them securely in object storage, and enqueue extraction tasks asynchronously. 

By decoupling file ingestion from heavy computational processing (OCR, LLM parsing, embedding generation), the service ensures high availability and resilience under load. It prevents API rate-limiting issues on downstream LLM providers and guarantees that no uploaded candidate profile is ever lost.

## 2. Target User

### 2.1 Jobs To Be Done (JTBD)
* **Functional:** As a recruiter, I want to upload a batch of 100 resumes at once and see their processing status in real-time, so I can start matching candidates without waiting for individual file uploads to finish.
* **Functional/Technical:** As an engineer, I want the ingestion service to buffer incoming files using a queue, so that high traffic spikes do not crash the system or exceed external LLM rate limits.
* **Emotional:** As a recruiter, I want to trust that every file I upload is safely received and processed, so I do not worry about missing candidate profiles.

### 2.2 Key User Journeys

* **UJ-1. Recruiter uploads a candidate resume batch.**
  * **Persona + context:** HR recruiter bulk uploading resumes for a newly opened software developer role.
  * **Entry state:** Authenticated on the web dashboard, viewing the job matching panel for JD-101.
  * **Path:** Recruiter selects 10 PDF resumes from their local file system, drags them into the UI upload zone, and clicks "Process."
  * **Climax:** The web interface immediately updates, showing 10 progress bars labeled "Queued" with active tracking IDs. The user doesn't have to wait for extraction; they can navigate away or view other screens.
  * **Resolution:** In the background, Spring Boot stores the files, queues the messages in RabbitMQ, and updates the database state. A worker picks them up sequentially, parses them, and updates the UI status to "Processing" -> "Completed" within seconds.
  * **Edge case:** One of the uploaded files is a password-protected PDF. The system immediately rejects it at ingestion with a clear "File is password protected" message, preventing it from consuming resource time.

## 3. Glossary

* **CV (Curriculum Vitae)** — A candidate's resume document, submitted in PDF or DOCX format.
* **Ingestion Pipeline** — The sequence of steps starting from file upload, ending with storage, queue publish, OCR text extraction, LLM parsing, and pgvector storage.
* **Tracking ID** — A unique UUID generated by the Ingestion Service upon file receipt to identify the processing task.
* **RabbitMQ Message Broker** — The messaging middleware used to store and distribute parsing jobs asynchronously.
* **Consumer Prefetch Count** — The configuration setting (`basic.qos`) that limits the number of unacknowledged messages a RabbitMQ worker can pull from the queue, managing backpressure.
* **DLQ (Dead-Letter Queue)** — A RabbitMQ queue where failed or unprocessable messages are redirected for inspection or manual retrying.
* **LLM Worker** — The background worker component responsible for reading text, invoking the parsing LLM, and generating candidate embeddings.

## 4. Features

### 4.1 Asynchronous File Upload & Ingestion
**Description:** The system must accept resume file uploads, store the raw documents, register the ingestion job in the database, and publish a processing event to RabbitMQ. This operation must be non-blocking. Realizes UJ-1.

**Functional Requirements:**

#### FR-1: REST File Upload
The system must expose a REST endpoint accepting `multipart/form-data` uploads containing CV files (PDF/DOCX) and an optional job description identifier (JD ID).
* **Consequences (testable):**
  * HTTP status `202 Accepted` is returned for valid requests, along with a payload containing the `trackingId` and status `QUEUED` within 100ms.
  * HTTP status `400 Bad Request` is returned if the file is empty, exceeds size limits, or uses an unsupported extension.

#### FR-2: File Storage & Job Tracking
The service must store the uploaded file in object storage `[ASSUMPTION: MinIO or Local Disk for V1]` and create a record in the database with status `QUEUED`.
* **Consequences (testable):**
  * A file is present in the storage bucket at the path specified in the DB record.
  * A record exists in the database table `cv_processing_jobs` mapping `trackingId` to file metadata, status `QUEUED`, and created timestamp.

#### FR-3: RabbitMQ Task Enqueuing
Upon successful storage, the service must publish a message containing the processing metadata to the RabbitMQ exchange.
* **Consequences (testable):**
  * A message is published to the `cv.ingestion.exchange` using the routing key `cv.upload.success`.
  * If publishing fails, the database job record status is updated to `FAILED` with an error message, and a transaction rollback occurs.

### 4.2 Job Status & Progress Tracking
**Description:** Recruiter clients must be able to query the processing status of any uploaded CV.

**Functional Requirements:**

#### FR-4: Query Progress API
The service must expose a REST endpoint to query the status of a job using its `trackingId`.
* **Consequences (testable):**
  * Returns HTTP `200 OK` with JSON containing the status (`QUEUED`, `EXTRACTING`, `PARSING`, `COMPLETED`, `FAILED`), error details (if failed), and candidate details (if completed).
  * Returns HTTP `404 Not Found` if the `trackingId` does not exist.

## 5. API Contracts & Payload Schemas

### 5.1 REST API Contracts

#### 1. Ingest Resume
* **Endpoint:** `POST /api/v1/cv/upload`
* **Content-Type:** `multipart/form-data`
* **Request Parameters:**
  * `file`: MultipartFile (Binary PDF/DOCX, max 10MB)
  * `jdId`: UUID (String, Optional)
* **Response:**
  * **Status:** `202 Accepted`
  * **Body (JSON):**
    ```json
    {
      "trackingId": "9b1deb4d-3b7d-4bad-9bdd-2b0d7b3dcb6d",
      "status": "QUEUED",
      "fileName": "johndoe_resume.pdf",
      "fileType": "pdf",
      "timestamp": "2026-07-01T15:35:00Z"
    }
    ```
  * **Errors:**
    * `400 Bad Request` (Invalid format, missing file, or file exceeds size)
      ```json
      {
        "errorCode": "INVALID_FILE",
        "errorMessage": "Unsupported file format. Only PDF and DOCX files are allowed."
      }
      ```

#### 2. Get Job Status
* **Endpoint:** `GET /api/v1/cv/status/{trackingId}`
* **Response:**
  * **Status:** `200 OK`
  * **Body (JSON):**
    ```json
    {
      "trackingId": "9b1deb4d-3b7d-4bad-9bdd-2b0d7b3dcb6d",
      "status": "COMPLETED",
      "error": null,
      "updatedAt": "2026-07-01T15:35:15Z"
    }
    ```
  * **Errors:**
    * `404 Not Found` (Job ID not found)
      ```json
      {
        "errorCode": "JOB_NOT_FOUND",
        "errorMessage": "Job with tracking ID 9b1deb4d-3b7d-4bad-9bdd-2b0d7b3dcb6d does not exist."
      }
      ```

## 6. Error Handling Strategy & Resilience

To build a high-performance system, errors must be categorized and mitigated at every stage of the pipeline:

```
Upload request ──> [ File Validation ] ─(Fail)─> 400 Bad Request response
                          │
                        (Pass)
                          │
                    [ Storage / Queue ] ─(Fail)─> Database Job marked FAILED
                          │
                        (Pass)
                          │
                    [ Ingestion Queue ]
                          │
                   (Worker Pull)
                          │
                [ Apache Tika OCR ] ─(Timeout/Format Error)─> Retry (Max 3) ──> DLQ
                          │
                       (Success)
                          │
                [ LLM API Parsing ] ─(429 Rate / 5xx)───────> Retry (Exponential backoff) ──> DLQ
```

### 6.1 Ingestion-time Failures (Synchronous)
* **File Validation Errors:** Invalid extension, empty files, or files > 10MB are rejected on the REST controller immediately. The server returns HTTP 400.
* **Storage/Database Failures:** If file storage throws an exception or the database fails to write, the process is aborted. The server returns HTTP 500, no message is enqueued, and any temp storage is cleaned.

### 6.2 Processing-time Failures (Asynchronous)
* **Text Extraction (OCR) Timeout:** Apache Tika text extraction can hang on large scanned PDFs.
  * **Mitigation:** The extraction logic will execute under a strict thread execution timeout `[ASSUMPTION: Text Extraction Timeout = 30 seconds]`.
  * **Failure Action:** If extraction times out, the worker throws a custom `TextExtractionException` triggering a RabbitMQ retry.
* **LLM API Rate Limits (HTTP 429) & Downstream Outages (HTTP 5xx):**
  * **Mitigation:** The worker uses exponential backoff retries when making HTTP requests to the LLM API.
  * **Failure Action:** The exception is propagated as a retryable error, returning the message to RabbitMQ to try again after a delay.
* **Parsing Schema Failures (LLM returns malformed JSON):**
  * **Mitigation:** LLM responses are validated against a target JSON Schema before saving.
  * **Failure Action:** If validation fails, a retry is executed with a modified temperature prompt. If it fails again, the job is marked `FAILED` and sent to the DLQ.

### 6.3 RabbitMQ Retry and Dead-Letter Queue (DLQ) Configuration
* **Prefetch Control:** Workers are configured with a prefetch limit (`basic.qos(prefetch_count=2)`). This ensures a worker does not consume more jobs than it can handle, protecting system memory.
* **Retry Policy:**
  * Max delivery attempts: 3.
  * Backoff: Exponential (initial interval 2s, multiplier 2, max interval 10s).
* **DLQ Routing:**
  * Exchange: `cv.ingestion.dlx` (Type: Direct)
  * Queue: `cv.dlq`
  * Messages exceeding the max retry count are moved to `cv.dlq`. An alert is generated, and the database status of the corresponding job is updated to `FAILED` with a log containing the exception stack trace.

## 7. Non-Goals (Explicit)

* **Automatic Reparsing on JD updates:** If a Job Description changes, CVs already parsed will *not* be automatically re-parsed by the LLM in V1 (they can be re-evaluated via vector search).
* **File Encryption:** The service does not encrypt files at rest in object storage for V1.
* **Virus Scanning:** Implementing file anti-virus scanning on upload is deferred.

## 8. MVP Scope

### 8.1 In Scope
* REST API endpoints for file upload and status polling.
* Local filesystem storage mocking S3 bucket storage.
* RabbitMQ Integration with Topic Exchange and Retry/DLQ structure.
* Core database table structures in PostgreSQL for tracking job states (`QUEUED`, `EXTRACTING`, `PARSING`, `COMPLETED`, `FAILED`).
* Request validation for file types (PDF/DOCX) and file size (max 10MB).

### 8.2 Out of Scope for MVP
* AWS S3 object storage integration (uses local storage mock).
* Real-time notifications (WebSockets) for processing updates (polling is used instead).
* Multi-lingual OCR support (V1 will assume English/Vietnamese standard latin characters).

## 9. Success Metrics

* **Primary Metrics:**
  * **Ingestion Latency:** REST response for file upload is returned in `< 100ms` for 95% of requests.
  * **resilience Rate:** 100% of uploaded files are successfully stored and enqueued, with zero lost messages.
  * **Retry Rate:** Less than 2% of processing tasks fail due to transient timeouts or API exhaustion (rate limits) thanks to backoff retries.
* **Counter-Metrics:**
  * **LLM API Cost:** Do not optimize parsing speed if it increases LLM costs by using larger, more expensive models or redundant calls (maintain GPT-4o-mini).

## 10. Open Questions

1. **OCR Support:** Do we need custom OCR model integration (like Tesseract) for scanned resumes that Apache Tika cannot extract text from directly?
2. **Redis Task Locking:** Should Redis be used to implement distributed locks to prevent the same tracking ID from being processed twice in case of RabbitMQ network partition?

## 11. Assumptions Index

* `[ASSUMPTION: MinIO or Local Disk for V1]` (Section 4.1, FR-2) — Using local filesystem path to simulate cloud bucket storage.
* `[ASSUMPTION: Text Extraction Library]` (Section 3 & 5.2) — Apache Tika is chosen for extracting text from PDF/DOCX.
* `[ASSUMPTION: Parsing LLM]` (Section 3 & 5.2) — Gemini 2.5 Flash is chosen (via OpenAI compatible REST API) for parsing raw resume text into structured fields.
* `[ASSUMPTION: Embedding Model]` (Section 3 & 5.2) — Gemini `text-embedding-004` (configured at 768 dimensions) is chosen for candidate embedding generation.
* `[ASSUMPTION: Prefetch Count]` (Section 6.3) — Consumer prefetch is set to 2.
* `[ASSUMPTION: Text Extraction Timeout = 30 seconds]` (Section 6.2) — OCR thread execution is timed out at 30 seconds.

## 12. Gemini API Integration & Vector Adaptation

This section specifies the integration details of Google Gemini API (using OpenAI-compatible endpoints) and the adaptation of database structures for 768-dimension vector storage.

### 12.1 Environment Configuration & Key Security
* The API Key for Gemini must be stored in the root `.env` file as `GEMINI_API_KEY`.
* Spring Boot properties will bind this variable: `app.openai.api-key: ${GEMINI_API_KEY}`.

### 12.2 Endpoint and Model Configurations
* **Chat Completions (CV Parsing)**:
  * URL: `https://generativelanguage.googleapis.com/v1beta/openai/chat/completions`
  * Model: `gemini-2.5-flash`
* **Embeddings (Candidate Summary & Search Query)**:
  * URL: `https://generativelanguage.googleapis.com/v1beta/openai/embeddings`
  * Model: `text-embedding-004` (produces 768-dimension output)

### 12.3 Database Schema Migration (pgvector)
* Since Gemini `text-embedding-004` produces a **768-dimension** vector, the candidate profile embedding field must be changed from `vector(1536)` to `vector(768)`.
* Both Backend and Worker entity class files (`CandidateProfile.java`) must update the column definition annotation: `@Column(name = "embedding", columnDefinition = "vector(768)")`.
* Before running, the old `candidate_profiles` table must be dropped/re-created in PostgreSQL to avoid shape mismatch exceptions.

### 12.4 Real JSON Deserialization (Jackson ObjectMapper)
* The worker service's `GptParserServiceImpl.java` will use Jackson `ObjectMapper` configured with `DeserializationFeature.FAIL_ON_UNKNOWN_PROPERTIES = false` to deserialize the structured LLM response into `ParsedCandidateDto`, eliminating hardcoded placeholder values.
